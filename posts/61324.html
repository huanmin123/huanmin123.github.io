我们先来介绍 4拦截方式

  1. 使用xml的方法 或者使用java配置类的方式 来管理 访问权限

        <security:intercept-url pattern="/**" access="ROLE_USER" />

    这种方式 太笨重了 每次都要在 项目里的配置文件中修改

  2. 将 路径 存放在数据库中 然后利用自定义拦截器来管理 访问权限

    这种方案比较不错 基本上开发都在使用 可以任意给角色加权限 和更换权限很灵活

    但是还是有点不足的是如果是公有的页面那么 我想要所有角色都能访问到 那么就需要一 一给每个角色 配置这个公有页面的访问路径 如果有100个角色呢?这不要累死. 否则就会被自定义的拦截器给拦截了 当然你可能会想 我将没有找到的路径直接放行可以了 只限制数据库里以及存在的路径 这样也行的 但是有更好的方式为什么不用呢

  1. 使用注解的方式

    将注解使用在方法上 一般都是作用在Controller层的 也有人作用在Service层看个人

    这样在编写代码的时候 就对指定的方法进行了限制 不需要在将路径录入进数据库里了 这样好处是方便 但是后期不易更换 所以一般都是配和自定义拦截器 公有的使用注解直接放行 而关键的页面使用数据库保存着就给指定的角色 随时更换

  1. 在jsp页面中使用标签的方式

    这种方式用的不多 一般都是在页面某些内容只能给 指定的角色看的时候使用 如果权限不符合 那么被权限 标签 包住的 内容 是不显示的.

以上这4种权限处理是有 顺序的

自定义拦截器和xml的方式是在 使用注解和jsp标签之前 只有当他们放行了后注解和jsp才有用 因为他们不放行 直接就403了 还怎么到方法层??或者jsp页面中

所以在自定义拦截器里我们要将 数据库中没有找到的路径放行 交给后面的管理而不是直接拦截下来

使用权限管理的时候先在配置中添加 自定义403页面 发生403无权限的时候跳转的地方

      <!--        如果发送无权限 的情况自动跳转到 403 自定义页面-->
        <security:access-denied-handler error-page="/html/403.html"/>

1.xml拦截方式

  <security:intercept-url pattern="/index.jsp" access="IS_AUTHENTICATED_ANONYMOUSLY" />
        <security:intercept-url pattern="/html/login.html" access="IS_AUTHENTICATED_ANONYMOUSLY" />
        <security:intercept-url pattern="/**" access="ROLE_USER" />

pattern: 拦截的url

access:表示需要什么角色才能访问pattern

IS_AUTHENTICATED_ANONYMOUSLY 表示用户不需要登录就可以访问

如我们通常会将登录地址和主页设置为上面的

IS_AUTHENTICATED_FULLY 用户需要是通过登录入口进行登录认证的才能访问

注意书写顺序 就和if else if 一样 找到符合后就不会 继续往下找了 也就是从小到大写 最后直接将所有路径拦截 只能ROLE_USER角色才能访问 也就是超级管理员

2. 自定义拦截器

在控制器拦截前 将所有请求拦截 然后进行处理 符合要求的返回角色不符合要求的看情况处理是直接拦截住 还是放行 取决你自己

还有就是 你要是用户 -角色和URL资源关系没 弄明白的 就别先玩 这个自定义拦截器 给你绕懵逼 你先去使用xml拦截方式和注解拦截方式以及jsp便签拦截方式

1.获取请求的url和数据库中的对比

这个类主要修改 loadAuthorityResources这个方法里的初始化数据 要和你的数据库一致 其他的地方基本上都有注释 看情况修改把 不懂就默认就行

package cn.xh.service.securityURL;

import cn.xh.dao.IPermissionDao;
import cn.xh.dao.IRoleDao;
import cn.xh.entity.Permission;
import cn.xh.entity.Role;
import cn.xh.utils.log4jutils;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.security.access.ConfigAttribute;
import org.springframework.security.access.SecurityConfig;
import org.springframework.security.web.FilterInvocation;
import org.springframework.security.web.access.intercept.FilterInvocationSecurityMetadataSource;
import org.springframework.stereotype.Service;
import org.springframework.util.AntPathMatcher;
import org.springframework.util.PathMatcher;

import java.io.File;
import java.util.*;

@Service("MFilterInvocationSecurityMetadataSource")
public class MFilterInvocationSecurityMetadataSource implements FilterInvocationSecurityMetadataSource {

    @Autowired
    private IPermissionDao url;
    @Autowired
    private IRoleDao Role;

    private  PathMatcher antPathMatcher = new AntPathMatcher();

    //  key  是url      value是角色     一个路径可以有多个角色所以是集合
    private Map<String, List<ConfigAttribute>> resources;
    //此方法是初始化   resources   将数据库里的角色 和角色对应的路径查询出来 放入resources中
    private void loadAuthorityResources() {
        resources = new HashMap<>();
        List<Permission> all = url.findAll();//此方法是查询所有 url 和相关数据
        for (Permission permission : all) {
            log4jutils.logger.info(permission.getId());
            String permissionId=permission.getId();//通过ID来获取角色
            List<Role> roleB = Role.findRoleBypermissionId(permissionId);
            List<ConfigAttribute> authorityList = new ArrayList<>();
            //将url对应的所有角色存入到 authorityList中
            for (cn.xh.entity.Role role : roleB) {
                authorityList.add(new SecurityConfig(role.getRoleName()));
            }
            //将 URL  和  authorityList(角色)   添加到 resources中
            resources.put(permission.getUrl(), authorityList);
        }

        log4jutils.logger.info(resources);

    }

    //此方法就是  拦截路径的
    @Override
    public Collection<ConfigAttribute> getAttributes(Object object) throws IllegalArgumentException {
        loadAuthorityResources();  //调用初始化方法

        log4jutils.logger.info("getAttributes");

        String url = ((FilterInvocation) object).getRequestUrl();//获取用户请求的url
            log4jutils.logger.info(url);
            //遍历 resources
        for(Map.Entry<String, List<ConfigAttribute>> entry:resources.entrySet()){
            //左边匹配右边  左边数据库  右边url
            // ** 代表能匹配 路径参数   如果不加** 那么就是绝对匹配不能差一点
           String key_url =entry.getKey(); //获取url
            //判断是否加**
            if (url.contains("?")&&!url.endsWith(File.separator)&&!(key_url.contains("*")||key_url.contains("**"))){
                key_url+="**";
            }
            //判断数据库中的URL 和请求的URL 是否匹配
            if (antPathMatcher.match(key_url,url)) {
                log4jutils.logger.info(key_url);
                log4jutils.logger.info("拥有的角色"+entry.getValue());
                if (entry.getValue().size()==0){
                    break;
                }
                //将角色传递给 MyAccessDecisionManager  由他处理
                return entry.getValue();
            }
        }

        //以下两种没有找到 对应的URL 和角色 后的  拦截方式

        //这种拦截方式   没有找到对应的 权限 直接403
//        log4jutils.logger.info("没找对应的角色   默认需要ROLE_USER角色 才能访问此url");
        //如果没有匹配成功，就说明该请求需要拥有ROLE_USER角色就能访问， 返回登录页面。
//        return SecurityConfig.createList("ROLE_USER"); //拦截除了ROLE_USER角色外   全部路径请求

        //这种拦截方式可以配合 注解使用
        log4jutils.logger.info("没找对应路径的角色  那么此路径就是公有的  没有加权限");
        //如果没有匹配成功，就说明该请求没有加权限，
        return null;  //不拦截
    }

    @Override
    public Collection<ConfigAttribute> getAllConfigAttributes() {
        return null;
    }
    @Override
    public boolean supports(Class<?> clazz) {
        return true;
    }
}

2.获取返回来的角色进行判断

这个类 没有什么好修改的地方 基本都写死了

package cn.xh.service.securityURL;

import org.springframework.security.access.AccessDecisionManager;
import org.springframework.security.access.AccessDeniedException;
import org.springframework.security.access.ConfigAttribute;
import org.springframework.security.authentication.InsufficientAuthenticationException;
import org.springframework.security.core.Authentication;
import org.springframework.security.core.GrantedAuthority;
import org.springframework.stereotype.Service;

import java.util.Collection;
import java.util.Iterator;

@Service("MyAccessDecisionManager")
public class MyAccessDecisionManager implements AccessDecisionManager {
    /*
     * 该方法决定该权限是否有权限访问该资源，其实object就是一个资源的地址，authentication是当前用户对应权限角色
     * ，如果没登陆就为游客匿名用户ROLE_ANONYMOUS，登陆了就是该用户对应的权限
     */
    @Override
    public void decide(Authentication authentication, Object object,
                       Collection<ConfigAttribute> configAttributes)
            throws AccessDeniedException, InsufficientAuthenticationException {
        if (configAttributes == null) {
            return;
        }
        System.out.println(object.toString()); //  URL.
        //所请求的资源拥有的角色(一个URL对 多个角色)
        Iterator<ConfigAttribute> iterator = configAttributes.iterator();
        while (iterator.hasNext()) {
            ConfigAttribute configAttribute = iterator.next();
            //访问所请求资源所需要的权限
            String needPermission = configAttribute.getAttribute();
            System.out.println("访问" + object.toString() + "路径" + "需要的权限是：" + needPermission);
            //用户所拥有的权限authentication
            Collection<? extends GrantedAuthority> authorities = authentication.getAuthorities();
            for (GrantedAuthority ga : authorities) {
                System.out.println("用户所拥有的权限" + ga.getAuthority());
                if (needPermission.equals(ga.getAuthority())) {
                    System.out.println("有权限了");
                    return;  //放行
                }
            }
        }
        //没有权限    这个异常会触发403
        throw new AccessDeniedException(" 没有权限访问！");
    }

    @Override
    public boolean supports(ConfigAttribute attribute) {
        return true;
    }

    @Override
    public boolean supports(Class<?> clazz) {
        return true;
    }
}

3.重写拦截器

当然这个拦截器从不重写都行 重写还是好的便于日后 可以进行扩展

以下不用动 默认就行

package cn.xh.service.securityURL;

import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.security.access.SecurityMetadataSource;
import org.springframework.security.access.intercept.AbstractSecurityInterceptor;
import org.springframework.security.access.intercept.InterceptorStatusToken;
import org.springframework.security.web.FilterInvocation;
import org.springframework.security.web.access.intercept.FilterInvocationSecurityMetadataSource;
import org.springframework.stereotype.Service;

import javax.servlet.*;
import java.io.IOException;

//拦截器
public class MyFilterSecurityInterceptor extends AbstractSecurityInterceptor
        implements Filter {
    private FilterInvocationSecurityMetadataSource securityMetadataSource;


    //拦截方法
    public void doFilter(ServletRequest request, ServletResponse response,
                         FilterChain chain) throws IOException, ServletException {
        FilterInvocation fi = new FilterInvocation(request, response, chain);
        invoke(fi);
    }

    public FilterInvocationSecurityMetadataSource getSecurityMetadataSource() {
        return this.securityMetadataSource;
    }

    public Class<? extends Object> getSecureObjectClass() {
        return FilterInvocation.class;
    }

    public void invoke(FilterInvocation fi) throws IOException,
            ServletException {
        InterceptorStatusToken token = super.beforeInvocation(fi);
        try {
            fi.getChain().doFilter(fi.getRequest(), fi.getResponse());
        } finally {
            super.afterInvocation(token, null);
        }
    }

    public SecurityMetadataSource obtainSecurityMetadataSource() {
        return this.securityMetadataSource;
    }

    public void setSecurityMetadataSource(
            FilterInvocationSecurityMetadataSource newSource) {
        this.securityMetadataSource = newSource;
    }

    public void destroy() {
    }

    public void init(FilterConfig arg0) throws ServletException {
    }

}

5.配置xml

然后就到了关键的地方配置spring-security.xml配置文件

 <security:authentication-manager alias="authenticationManager">
        <!-- 1、配置自定义类User_verify -->
        <security:authentication-provider   user-service-ref="User_verify"  >
            <!-- 配置加密的方式   对登录的password数据进行加密 否则和数据库的密码不匹配
            因为在添加用户的时候 密码也是加密的  -->
            <security:password-encoder  ref="passwordEncoder"/>
        </security:authentication-provider>
    </security:authentication-manager>
<!--   security自带的 加密类   不可逆SHA-256算法-->
    <bean id="passwordEncoder" class="org.springframework.security.crypto.bcrypt.BCryptPasswordEncoder"/>


    <bean id="filterSecurityInterceptor" class="cn.xh.service.securityURL.MyFilterSecurityInterceptor">
        <!-- 用户拥有的权限 -->
        <property name="authenticationManager" ref="authenticationManager" />
        <!-- 用户是否拥有所请求资源的权限 -->
        <property name="accessDecisionManager" ref="MyAccessDecisionManager" />
        <!-- 资源与权限对应关系 -->
        <property name="securityMetadataSource" ref="MFilterInvocationSecurityMetadataSource" />
    </bean>

其他的估计都能看懂 估计 user-service-ref=”User_verify”不知道什么意思把

这个是我自定义的 用户登录处理类 可以在我博客中找到

还有就是<security:password-encoder ref=”passwordEncoder”/>这个是针对用户登录的时候进行解密和密码对比 因为我们注册的时候 密码都是加密后存入数据库里的 我们需要将输入的密码和数据库中的密文进行比对

然后 添加

<!--自定义拦截器 -->
        <security:custom-filter ref="filterSecurityInterceptor" before="FILTER_SECURITY_INTERCEPTOR" />

将这个放入到 xml中 要在放在所有 xml拦截器的 前面 当然此时所有URL权限拦截 此时就可以删除了 没什么用了 我们基本在自定义里都处理完了 也就是这个<security:intercept-url 将自定义拦器放在所有此标签的前面

之后就可以拦截了

3.注解的方式拦截

使用注解的准备

我们需要导入Maven jsr250-api 这个jar 我们上面已经导入过了

使用注解前需要在spring-security.xml配置文件中开启支持的


    <!--    开启权限管理的一系列注解支持  JSR-250注解、@Secured注解和支持表达式的注解-->
    <security:global-method-security  jsr250-annotations="enabled" secured-annotations="enabled"  proxy-target-class="true" pre-post-annotations="enabled"/>

如果发现注解没有生效 那么你还要在MVC配置文件中添加

    <!--    使用 spring-security 注解支持 需要在MVC中开启 <aop:aspectj-autoproxy
     不是applicationContext.xml 里面已经有了吗 怎么还开启
     这个我也不懂 但是如果不写 那么注解就没用
    -->
    <aop:aspectj-autoproxy proxy-target-class="true"/>

如果你不知道AOP约束信息的话这里提供 完整版MVC约束 直接替换就行

如果在使用注解的过程中没有发送403权限异常然后进行跳转到自定义403页面中

而是发生 Access is denied 没有权限的意思 你可以定义一个MVC拦截器 这个错误是RuntimeException运行时错误 然后进行拦截 然后跳转到403页面

@ControllerAdvice  //异常拦截
public class ControllerExceptionResolver {
        @ExceptionHandler(RuntimeException.class)
    public  ModelAndView RuntimeException(RuntimeException exception, HttpServletRequest request ){
        ModelAndView md=new ModelAndView();
        String message = exception.getMessage();
        log4jutils.logger.info("  @ExceptionHandler(RuntimeException.class)");
        log4jutils.logger.info( exception.getMessage());

        //如果遇到错误是 没用权限那么 就跳转到 权限页面
        if (message.trim().equals("Access is denied")){
            md.setViewName("redirect:/jsp/403.jsp");
            return md;
        }
        md.addObject("message",message);
        md.setViewName("forward:/jsp/500.jsp");
        return md;
    }

}

4.jsp标签拦截方式

在jsp头部添加

<%@taglib uri="http://www.springframework.org/security/tags" prefix="security"%>

还需要在Maven中导入spring-security-taglibs这个jar 我们上面已经导入过了

主要就那几个标签 用处就是将 内容隐藏 只给权限够的人看


<%@taglib uri="http://www.springframework.org/security/tags" prefix="security"%>
<%@ page contentType="text/html;charset=UTF-8" language="java" %>
<html>
<head>
    <title>Title</title>
</head>
<body>
<%--property="principal" 必须的   var 将用户权限信息 存储在authentication中  scope作用域  --%>
<security:authentication property="principal" var="authentication" scope="request" />
用户名:${authentication.username}<br>

<security:authorize access="hasAnyRole('ROLE_SHOW','ROLE_ADMIN')">
        <br>用户管理 'ROLE_SHOW','ROLE_ADMIN'
</security:authorize>

</body>
</html>

如果发现标签 没有效果 那么肯定是在xml中<security:http security="none" pattern="/jsp/**" />给放行了 因为这个标签声明的文件内的东西 是不会被拦截的 所以不会被权限标签进行处理的 所以内容一直隐藏 默认的就是隐藏 满足权限才显示 但是我都给他放行了 权限拦截器拦截不到内容 所有也就无效了

还有就是 你定义了自定义拦截器那么就不用加这条 如果没有自定义拦截 也没放行此页面 权限标签还是无效 那么就将这条加在spring-security.xml里 放在所有<security:http标签的前面

    <bean id="webSecurityExpressionHandler" class="org.springframework.security.web.access.expression.DefaultWebSecurityExpressionHandler" />
