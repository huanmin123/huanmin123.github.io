java多线程之-CAS乐观锁

什么是悲观锁、乐观锁？在java语言里

  • synchronized是悲观锁，这种线程一旦得到锁，其他需要锁的线程就挂起的情况就是悲观锁。
  • CAS操作的就是乐观锁，每次不加锁而是假设没有冲突而去完成某项操作，如果因为冲突失败就重试，直到成功为止。

在进入正题之前，我们先理解下下面的代码:

    static   Integer com=0;
    public static void main(String[] args) throws InterruptedException {
        int num=3;
        for (int i = 0; i < num; i++) {
            new  Thread(new Runnable() {

                @Override
                public void run() {
                    for (int j=0;j<10000;j++){
                            com++;
                    }
                }
            }).start();
        }
        Thread.sleep(100);
        System.out.println(com);
    }

请问 com的输出值是否为30000？答案是否定的，因为这个程序是线程不安全的，所以造成的结果count值可能小于30000;

那么如何改造成线程安全的呢，其实我们可以使用上Synchronized同步锁,我们只需要在com++的位置添加同步锁，代码如下:

  static   Integer com=0;
    public static void main(String[] args) throws InterruptedException {
        int num=3;
        for (int i = 0; i < num; i++) {
            new  Thread(new Runnable() {

                @Override
                public void run() {
                    for (int j=0;j<10000;j++){
                        synchronized (CAS_Int.class){
                            com++;
                        }
                    }
                }
            }).start();
        }
        Thread.sleep(100);
        System.out.println(com);
    }
}

加了同步锁之后，com自增的操作变成了原子性操作，所以最终的输出一定是30000，代码实现了线程安全

但是Synchronized虽然确保了线程的安全，但是在性能上却不是最优的，Synchronized关键字会让没有得到锁资源的线程进入阻塞状态，而后在争夺到锁资源后进入运行状态，这些操作代价比较高。

尽管Java1.6为Synchronized做了优化，增加了从偏向锁到轻量级锁再到重量级锁的过度，但是在最终转变为重量级锁之后，性能仍然较低。

所谓原子操作类，指的是java.util.concurrent.atomic包下，一系列以Atomic开头的包装类。例如AtomicBoolean，AtomicInteger，AtomicLong,AtomicReference。它们分别用于Boolean，Integer，Long,对象,类型的原子性操作。

这些类型的使用网上一大堆这里就不介绍了

      static    AtomicInteger atomicInteger=new AtomicInteger();
    public static void main(String[] args) throws InterruptedException {
        int num=3;
        for (int i = 0; i < num; i++) {
            new  Thread(new Runnable() {

                @Override
                public void run() {
                    for (int j=0;j<10000;j++){
                        //实现++操作
                        atomicInteger.incrementAndGet();
                    }
                }
            }).start();
        }
        Thread.sleep(100);
        System.out.println(atomicInteger);
    }
}

使用AtomicInteger之后，最终的输出结果同样可以保证是30000。并且在某些情况下，代码的性能会比Synchronized更好。而Atomic操作的底层实现正是利用的CAS机制.

Cas也不是完美无缺的

  1. 在并发量大的情况下，如果线程反复尝试更新变量却不能成功，导致不断循环会给cpu造成很大的压力

    ​ 为什么会不断循环 来看一下incrementAndGet()底层带码

        public final int getAndAddInt(Object var1, long var2, int var4) {
        int var5;
        do {
            var5 = this.getIntVolatile(var1, var2);
        } while(!this.compareAndSwapInt(var1, var2, var5, var5 + var4));

        return var5;
    }

    只有compareAndSwapInt返回为true 也就是添加成功后 才结束循环 怎么才能添加成功 就是当前预存值和主内存值相同的时候 如果在+1过程中主内存值被其他线程修改了那么就会添加失败然后循环继续..

  2. Cas只能同步一个变量，而不能同步一段代码，如果要对多个变量同时进行更新cas就无能为力了。

  3. Cas会造成ABA问题。

什么是CAS机制

CAS是英文单词Compare And Swap的缩写，翻译过来就是比较并替换。

CAS机制当中使用了3个基本操作数：主内存值V，预期值A，要修改的新值B。

更新一个变量的时候，只有当变量的预期值A和主内存值V相同时，才会将主内存值V值修改为B。 否则就继续尝试

什么是ABA问题？怎么解决？

当一个值从A更新成B，又更新会A，普通CAS机制会误判通过检测。

我们举一个提款机的例子。假设取款机遵守CAS协议的, 小灰有100元存款，要用这个提款机来转账50元

由于提款机出了点小问题,卡主了 ,小灰没时间等待，又重新找一个取款机，两个线程都是获取当前值100元，要更新成50元。

理想情况下，应该一个线程更新成功，另一个线程更新失败，小灰的存款只被扣一次。

线程2执行成功，把余额从100改成50。线程1因为卡主了还在阻塞状态。这时候，小灰的妈妈刚好给小灰汇款50元。

线程2仍然是阻塞状态，线程3执行成功，把余额从50改成100。

线程2恢复运行了，由于阻塞之前已经获得了当前值100，并且经过compare检测，此时存款实际值也是100，所以成功把变量值100更新成了50。

原本线程2应当提交失败，小灰的正确余额应该保持为100元，结果由于ABA问题提交成功了。

利用版本号比较可以有效解决ABA问题

使用 AtomicStampedReference<>

  static   AtomicInteger atomicInteger=new AtomicInteger(0);

    //版本是，以后每修改一次都会增加1，如果修改时和自己预料的版本不一致就会修改失败
    static AtomicStampedReference<Integer> atomicStampedReference = new AtomicStampedReference(null,0);

    public static  void AtomicABA(){
        int oldStamp = atomicStampedReference.getStamp(); //获取初始化版本号
        Integer oldReference = atomicStampedReference.getReference();//获取初始化值
         Integer newReference = atomicInteger.incrementAndGet(); //自增  CAS
                                                                 ///老值    新值        旧版本号           新版本号
          boolean  b =atomicStampedReference.compareAndSet(oldReference,newReference,oldStamp,oldStamp+1);
          //当旧修版本号 和之前的版本号相同没有被修改 那么就将老值修改为新值

    }
    public static void main(String[] args) throws InterruptedException {

        int num=3;

        for (int i = 0; i < num; i++) {
            new Thread(new Runnable() {
                @Override
                public void run() {
                   for (int j=0;j<10000;j++){
                       AtomicABA();
                   }
                }
            }).start();
        }

        Thread.sleep(100);
        System.out.println(atomicStampedReference.getReference());//获取值
    }
